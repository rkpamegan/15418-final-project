## 5. 关键发现 / 原因分析（→ 报告）

### 5.1 Graph coloring 是绝对瓶颈（85% 时间）

**事实：** color phase 在 test2 / test1 上均占 ~85.5%，且 test2→test1（4× verts）时 color 时间只增长 1.26×。

**原因：** Jones-Plassmann 算法的当前实现：
- 每个 outer iter 调用 **3 次** coloring（split 前 1 次 edge coloring + collapse 前 1 次 edge re-coloring + smooth 前 1 次 vertex coloring）
- 每次 coloring 是一个 **host-driven loop**：
  ```
  while (!h_done) {
      h_done = true;
      cudaMemcpy(d_done, &h_done, ..., HostToDevice);
      kernel_color_*<<<...>>>(...);
      CUDA_CHECK("color_*");          // 含 cudaDeviceSynchronize
      cudaMemcpy(&h_done, d_done, ..., DeviceToHost);
  }
  ```
- 每轮 fix-point 迭代 = **2 次 cudaMemcpy + 1 次 kernel launch + 1 次 device sync**。Jones-Plassmann 通常需要 O(log n) 轮收敛 → 单次 coloring ≈ 10–20 个 round-trip。
- 3 phases × 3 outer iter = **9 次 coloring 调用 × ~15 round-trip = ~135 host↔device 同步**。每次 sync 在 PCIe 上 ~10 µs 的常驻开销。
- **Coloring 本身的 GPU 工作量极少**（每 edge/vertex 仅一次比较）；瓶颈完全在 sync 而非计算。

**支撑证据：** test1 (3458v) vs test2 (866v) 的 color 时间 206.87 / 164.57 = 1.26×，远小于顶点数比 4×。说明时间被 ~固定的 sync 开销主导，而非算力。

### 5.2 几何 kernel 扩展性良好

split + collapse + flip + smooth 合计：
- test2: 27.42 ms
- test1: 34.16 ms
- 比值 1.25×，相对 4× 顶点增长属**亚线性**（受益于 GPU 并行度足够）。

这表明：**只要解决 coloring，整个 pipeline 几乎是 GPU-throughput-bound，可以继续向更大 mesh 扩展。**

### 5.3 split 比 collapse 便宜得多

test1：split 3.12 ms vs collapse 18.08 ms（5.8×）。
原因：collapse 需要遍历两端顶点的 1-ring 并 rewire（带 1024-step fan-walk cap），而 split 几乎是常数工作量（1 vertex → 3 edges + 6 halfedges + 2 faces，无邻居遍历）。

### 5.4 Mesh 增长

test1 经 3 iter 后累计 split = 1510+3638+7216 = **12 364 个新顶点**（原 3458 → 终态 ~15 800）。所有现存 invariant guard（INVALID_IDX、1024 fan-walk cap、collapse C-rewire）在这种规模下仍保持正确，说明前期所有 bug 修复站得住脚。

## 7. 已修复的 bug 清单（accumulated）

| # | 模块 | 现象 | 修复 |
|---|---|---|---|
| 1 | collapse | vA/vB 用了 `hp_twin/tn_twin` 应为 `hn_twin/hp_twin` | 改写选择 |
| 2 | smooth | 邻居 walk 不检查 INVALID_IDX → 越界 | 全路径 INVALID guard |
| 3 | split | 顶点 buffer 没扩容 → 越界写 | realloc |
| 4 | collapse | 合并 edge 的 `halfedge_idx` 指向被释放 he | 重定向到幸存 he |
| 5 | get_edge_lengths | 无 INVALID guard | 加 guard |
| 6 | smooth | `vertex_normals` 未初始化（NaN） | （早期）`cudaMemset 0` workaround → 后续 (#11) 真正实现 |
| 7 | smooth | `vertex_pos` 没初始化（垃圾值被 update 写回） | `kernel_init_vertex_pos` |
| 8 | 全体 | 各 kernel 缺 INVALID_IDX guard | 系统补齐 |
| 9 | smooth/collapse | fan-walk 无环检测 → 死循环 | 1024-step hard cap |
| 10 | collapse | C-rewire walk 不验证 vC | 加 vC validation |
| 11 | normals | `kernel_get_vertex_normals` do-while 没推进 `curr_idx` | 重写 walk + INVALID guards + 1024 cap + zero-len NaN 防护 |
| 12 | normals 集成 | 主循环里没有调用 normals kernel | 在 smoothing 内层每 iter 调用 |
| 13 | mesh_from_file | `id_to_idx` 单 map，但 raw id 在 h/v/e/f **跨类型重叠** → 索引错乱 | 拆成 4 个 per-type map |
| 14 | mesh_from_file | 第二轮 remap 用值拷贝 `Mesh::X x = mesh->Xs[i]`，写回丢失 | 改为引用 `Mesh::X&` |
| 15 | mesh_from_file | 空行 `substr(2, ...)` 越界 → `std::out_of_range` | 跳过 `< 4` 字节或不以 `[` 开头的行 |
| 16 | main | `free(mesh)` 但 `mesh` 是 `new` 的 → UB | 改 `delete mesh;` |
| 17 | repo | `*.o`、`remesh` 被提交 → 每次 pull 冲突 | `.gitignore` 排除 + `git rm --cached` |

## 9. 下一步（写报告时可作为 Future Work / Optimization 段）

按预期收益排序：

1. **device-resident coloring**：把 fix-point loop 整体搬进 kernel（grid sync 或 cooperative groups），消除 ~135 个 host↔device round-trip。预期 color 时间 −5×~10×。
2. **CPU baseline 对比**：跑 `src/old/mesh-local.cpp` 同一组数据集，给出 speedup 数。
3. **更大 mesh / 多 outer iter**：当前 3 iter 不足以把 average length 压到目标，可跑 10–20 iter 看终态。
4. **写出 remesh 后 mesh**：加 `mesh_to_file()`，离线可视化质量。
5. **去掉 file-scope 死全局**：消除 cudaRemesh.cu:48-49 警告。
